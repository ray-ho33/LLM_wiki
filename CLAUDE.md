# LLM Wiki — 프로젝트 관리 지식 베이스

이 디렉토리는 LLM이 유지·관리하는 마크다운 위키입니다.
아래 규칙을 따라 위키를 운영하세요.

---

## 구조

```
LLM_wiki/
├── CLAUDE.md          ← 이 파일. 위키 운영 규칙.
├── .mcp.json          ← kordoc MCP 서버 설정
├── raw/               ← 원본 소스 (불변, 절대 수정하지 않음)
│   └── assets/        ← 이미지, 첨부파일
├── wiki/              ← LLM이 관리하는 위키 페이지
│   ├── index.md       ← 전체 페이지 카탈로그
│   ├── log.md         ← 작업 로그 (시간순)
│   ├── overview.md    ← 위키 현황 대시보드
│   ├── sources/       ← 소스별 요약
│   ├── projects/      ← 프로젝트별 페이지
│   ├── entities/      ← 조직, 인물, 시스템 등
│   ├── concepts/      ← 주요 개념, 용어 정의
│   ├── reports/       ← 분석, 비교, 질의 결과
│   └── issues/        ← 이슈, 리스크, 의사결정
└── tools/             ← (선택) 검색 스크립트 등
```

### 핵심 원칙

- `raw/` 는 **읽기 전용**. 절대 수정하지 않는다.
- `wiki/` 는 **LLM이 전적으로 관리**. 페이지 생성, 수정, 삭제 모두 LLM이 한다.
- 사용자는 소스를 `raw/`에 넣고, 질문을 하고, 방향을 잡는다. LLM이 나머지를 한다.

## 페이지 컨벤션

### Frontmatter

모든 위키 페이지는 YAML frontmatter로 시작한다:

```yaml
---
title: 페이지 제목
type: source | project | entity | concept | report | issue
tags: [태그1, 태그2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [관련 소스 파일명]
status: active | completed | archived  # project, issue에만 사용
---
```

### 파일 이름

- 한글 사용 가능. 공백 대신 언더바(`_`) 사용.
- 예: `wiki/projects/신규_시스템_구축.md`, `wiki/entities/기획팀.md`

### 내부 링크

- `[[페이지명]]` 형식 사용 (Obsidian 호환).
- 다른 폴더의 페이지: `[[projects/신규_시스템_구축]]`
- 링크 생성 시, 대상 페이지가 없으면 만든다.

### 태그 체계

소스 문서의 성격에 따라 태그를 부여한다. 예시:
- 문서 유형: `보고서`, `회의록`, `계획서`, `현황표`, `지침`, `공문`
- 업무 분야: 프로젝트/도메인에 따라 자유롭게 추가
- 상태: `진행중`, `완료`, `보류`

## 워크플로우

### 1. Ingest (소스 투입)

사용자가 `raw/`에 새 문서를 넣고 처리를 요청하면:

1. **파싱**: kordoc CLI 또는 MCP로 문서를 읽는다.
   - CLI: `kordoc <파일경로>` → 마크다운 출력
   - MCP: kordoc MCP 도구 사용
   - 이미지: Read 도구로 직접 열람
2. **사용자와 논의**: 핵심 내용을 요약하고, 어떤 점이 중요한지 사용자와 확인한다.
3. **소스 요약 페이지 작성**: `wiki/sources/` 에 요약 페이지 생성.
   - 문서 메타정보 (제목, 날짜, 출처, 유형)
   - 핵심 내용 요약
   - 주요 데이터/수치
   - 관련 위키 페이지 링크
4. **관련 페이지 업데이트**: 소스에서 언급된 프로젝트, 엔티티, 개념 등의 기존 페이지를 업데이트한다. 없으면 새로 만든다.
5. **인덱스 갱신**: `wiki/index.md`에 새 페이지들을 등록한다.
6. **로그 기록**: `wiki/log.md`에 작업 내역을 추가한다.
7. **overview 갱신**: `wiki/overview.md`의 현황 수치와 최근 활동을 업데이트한다.

### 2. Query (질의)

사용자가 질문하면:

1. **인덱스 탐색**: `wiki/index.md`를 읽고 관련 페이지를 식별한다.
2. **페이지 읽기**: 관련 페이지들을 읽고 정보를 종합한다.
3. **답변 생성**: 출처를 명시하며 답변한다. `[출처: [[페이지명]]]` 형식.
4. **(선택) 결과 저장**: 가치 있는 분석 결과는 `wiki/reports/`에 저장하고 인덱스에 등록한다.

### 3. Lint (건강 점검)

사용자가 점검을 요청하거나 주기적으로:

1. **모순 확인**: 페이지 간 상충하는 정보 찾기
2. **고아 페이지**: 다른 페이지에서 링크되지 않은 페이지 찾기
3. **빈 링크**: `[[...]]` 링크 중 대상 페이지가 없는 것 찾기
4. **오래된 정보**: 오랫동안 업데이트되지 않은 페이지 확인
5. **누락된 페이지**: 자주 언급되지만 전용 페이지가 없는 개념/엔티티
6. **데이터 갭**: 추가 소스가 필요한 영역 제안
7. 결과를 보고하고, 사용자 승인 후 수정한다.

## kordoc 사용법

### CLI로 파싱
```bash
# 단일 파일 → stdout에 마크다운 출력
kordoc raw/보고서.hwpx

# 특정 페이지만
kordoc raw/문서.pdf --pages 1-3

# 디렉토리 내 모든 문서 변환
kordoc raw/*.hwpx --out-dir wiki/sources/
```

### 지원 형식
- HWP, HWPX → 마크다운
- PDF (텍스트/OCR) → 마크다운
- XLSX → 마크다운 테이블
- DOCX → 마크다운

## index.md 관리 규칙

각 항목은 한 줄로 작성:
```
- [[sources/소스_파일명]] — 핵심 한줄 설명 (YYYY-MM-DD)
```

카테고리별로 정리하고, 새 페이지 추가 시 해당 카테고리에 삽입한다.
200개 이상이 되면 카테고리별 하위 인덱스 파일로 분리한다.

## log.md 관리 규칙

각 항목은 다음 형식:
```
## [YYYY-MM-DD] 작업유형 | 제목
- 수행한 작업 내역
- 업데이트한 페이지 목록
```

작업유형: `ingest`, `query`, `lint`, `update`, `init`
최신 항목이 아래로 추가된다 (append-only).
